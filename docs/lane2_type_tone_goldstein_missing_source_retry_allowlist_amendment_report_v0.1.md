## Retry/backoff design

- Bounded transient retry/backoff wraps each source-file read **before** the file
  enters the streaming buffer (new `fetch_resilient` → `_read_once_with_retry`).
- Retryable transient classes (`classify_fetch_exception`, value-blind, no network
  import at module load — matched by HTTP status and by exception-MRO class name):
  `http.client.IncompleteRead`, timeout errors (`TimeoutError`/`socket.timeout`),
  connection reset/abort (`ConnectionResetError`/`ConnectionAbortedError`/`ConnectionError`),
  transient `urllib` transport (`URLError`/`RemoteDisconnected`), **HTTP 429**, and
  **HTTP 5xx** (`{500,502,503,504}`).
- Bounded budget `max_transient_retries` (default 4) with exponential backoff
  `retry_backoff_base_seconds * 2^(n-1)` (default base 2.0s). Budget exhaustion →
  terminal `TransientFetchError` (no silent tolerance).
- **Retry never masks integrity failures:** MD5 / byte-size / exact-58 / stability
  checks are raised *after* a successful fetch, outside the retry try-block. A
  first-try success path performs no retry and no backoff sleep, so output bytes
  are unchanged (byte-neutral). Tests inject a fake `sleep_fn`; no test sleeps in
  real time.

## Stable-404 classification design

- A `404` is **not** an ordinary transient. The first 404 triggers a small bounded
  stability re-check (`_confirm_404_stability`, `stable_404_recheck_count` default 1
  — the code analogue of the probe's repeat-after-delay):
  - every re-check also 404 → **stable** → `ManifestPresentButUnavailableError`
    (manifest-listed-but-unavailable);
  - a re-check returns bytes → **recovered** → those bytes are used;
  - a re-check raises a transient → inconclusive → handled by the transient retry
    budget.
- A stable 404 is **never silently tolerated by itself**: it propagates to the
  orchestrator, which routes it to allowlist classification. Tests use fake
  `urllib.error.HTTPError(code=404)` fetchers; no real-time sleep.

## Allowlist config surface and default behavior

- **Surface:** `run_bounded_integrity_build(..., fetch_absent_allowlist=<seq of
  ISO YYYY-MM-DD source dates>)`, parsed by `parse_fetch_absent_allowlist`.
- **Default:** parameter default is `None` → empty/strict set. With an empty
  allowlist **every** stable-404 on a manifest-listed file is a terminal hard-fail
  (unchanged from pre-amendment unknown-404 behaviour).
- The allowlist is operator-supplied **per authorized run only**. It is never
  inferred from manifests, cache state, runtime failures, prior fail reports, or
  probe reports, and runtime failures never auto-populate it.
- A future authorized real rerun would pass it explicitly, e.g.
  `run_bounded_integrity_build(..., fetch_absent_allowlist=["2022-11-10"])`.

## Coverage propagation rule

A confirmed allowlisted fetch-absent source date is:
1. removed from the available-source set **during the loop**, before any dependent
   block is released; and
2. folded into the **same** coverage gap set used for mutual-absence gaps.

Both the streamed row-filter and the per-SQLDATE coverage ledger then route through
the **existing** production oracle (`fetched_set_fully_covers` /
`classify_sqldate_coverage`) — there is **no parallel special-case primary-set
filter** that could drift. Every SQLDATE whose `{t-1, t, t+1}` support includes a
fetch-absent date is stranded (gap-uncovered, cause named).
